# Бриф проекта: Telegram-бот для любителей бадминтона (v2)

> Рабочий документ для продолжения обсуждения. Фиксирует принятые решения и открытые вопросы,
> чтобы можно было возобновить работу в новом чате.
> Статус: концепция / проектирование. Кода пока нет.

## 1. Видение продукта

Бот-помощник для любительского бадминтона (комьюнити ЛАБ). Три слоя развития:

1. **Справочно-аналитический** (пассивный): игроки, турниры, история встреч, рейтинги, «сыгранность».
2. **Рекомендательный**: подбор партнёров, граф связанности игроков/турниров.
3. **Лайв-помощник** (активный): просчёт сетки на турнире, ведение счёта по партиям.

Развитие от слоя 1 к слою 3. Слои 1–2 питаются от внешних источников (парсинг), слой 3 генерирует
собственные данные во время игры.

**Roadmap:** MVP = слой 1 (H2H) → слой 2 (подбор партнёров) → слой 3 (лайв).

## 2. Источник данных

### Основной: `badminton4u.ru`
- Серверный рендер HTML, стабильные числовые ID. Не SPA, без капчи — удобно для парсинга.
- Парные данные (соперники/результаты) на страницах игрока отдаются **динамически** (вкладки/AJAX).

Карта источника:

| Данные | URL | Содержимое |
|---|---|---|
| Список турниров | `tournaments/?cities[]=..&types[]=..&date_from=..&date_to=..&winners=1` | фильтры по городам/разрядам/датам; средний рейтинг (`<var>`), категория в названии (SE, DF, XDC…), число участников |
| Турнир (будущий) | `tournaments/{id}` | список зарегистрированных игроков/пар |
| Турнир (прошедший) | `tournaments/{id}` | итоговая таблица: место, игрок(и), город, рейтинг до/дельта/после, матчи (W-L), сеты (W-L), коэффициент. Детальная сетка — **только картинками** (`res-N.png`) |
| Игроки | `players/` | справочник, фильтр по полу; ссылки на клуб/тренера/стрингера |
| Профиль игрока | `players/{id}` | ник, ФИО, город, дата рождения, рука, рейтинги по 7 дисциплинам (s, d, ms, ws, md, wd, xd), **история рейтинга с датами** (JS-массивы) |
| Соперники (H2H) | `rivals/{id}` | по каждому сопернику: число встреч, побед, поражений, дельта; ссылка на историю встреч |
| История встреч | `games/?user1ID=X&user2ID=Y` | список матчей: дата, турнир, стадия, **счёт по сетам (2:0)**, дельта рейтинга, длительность |
| Клубы/статистика | `halls/{id}`, `statistics/*` | клубы, агрегаты |
| Документация рейтинга | `content/2`, `content/10` | формула одиночного и парного рейтинга |

### Второй источник (позже, для лайв-слоя): `badminton77.ru`
Календарь турниров Москвы/МО и крупных соревнований РФ. Отложен.

### Стратегия сбора
- **Обход (seed от турниров):** турниры региона → участники (обнаружение ID игроков) → профили → соперники (сводка H2H). `games` тянем **лениво** по запросу пользователя.
- **Охват:** полный слепок по региону. Старт — **Москва и МО (`cities[]=r77`)**. Глубина — **последние 3 года**.
- **Частота обновления:** раз в сутки.
- **Этика:** вежливый парсинг — rate-limit, кеширование, честный User-Agent с контактом. При росте нагрузки — написать владельцам сайта про API/выгрузку.
- **При слепке храним:** сводку `rivals` для всех игроков региона; детальные `games` — лениво.

### Известные ограничения / риски
- Точный счёт **по очкам в партии** (21:15) в тексте недоступен — только счёт **по сетам** (2:0). Детальная сетка турнира — картинками (`res-N.png`).
- **pair-vs-pair история**: нативно данные — player-vs-player. Показывается ли партнёр в каждой парной встрече — доопределить на этапе парсера. Если нет — аппроксимировать pair-vs-pair через перекрёстные H2H четырёх игроков.

## 3. Официальные формулы ЛАБ (зеркалим as-is, не выдумываем)

- **Дельта рейтинга одиночки за матч:** `(100 - (R_поб - R_проигр)) / 10`.
  - Если рейтинг победителя ≥ рейтинга проигравшего на 100+ → дельта = 0 (победа «ожидаема»).
  - Симметрично: проигравший теряет столько же. Пол рейтинга = 1.
  - **Следствие:** разница ~100 пунктов ≈ решающее преимущество. Удобная шкала для прогноза.
- **Рейтинг пары:** `(A + B) / 2`, где A — рейтинг сильнейшего, B — слабейшего партнёра (источник: `content/10`).

## 4. Наши метрики (надстройки над данными)

### Индекс сыгранности S
```
S = Σ_i 0.5^(Δt_i / H)
```
Затухание по **каждой** встрече (Δt_i — время с i-й встречи, H — период полураспада).
Применяется к соперникам (H2H) и к партнёрам (совместные парные турниры).
H — **параметр конфигурации**.

### Форма игрока
```
Form = Σ_i δ_i · 0.5^(Δt_i / H)
```
**Сумма** взвешенных свежестью дельт ЛАБ `δ_i` (знаковых). Дельта кодирует ожидаемость результата.

### Прогноз P3 — явная формула

**Шаг 1. Эффективный рейтинг:**
```
R_eff_A = R_A + k · Form_A
R_eff_B = R_B + k · Form_B
```

**Шаг 2. Модель по рейтингу (логистика, масштаб 100):**
```
P_model = 1 / (1 + 10^(-(R_eff_A - R_eff_B) / 100))
```

**Шаг 3. Вероятность из личной истории (сглаживание Лапласа):**
```
P_h2h = (W_A + 1) / (W_A + W_B + 2)
```

**Шаг 4. Смешивание с весом индекса сыгранности:**
```
w = S / (S + S_ref)
P = (1 - w) · P_model + w · P_h2h
```

**Отображение:** «Фаворит: A (≈62%)» + краткое обоснование (рейтинг, форма, H2H, S).

Параметры `k`, `S_ref`, `H` — **в конфиге**, калибровка на реальных данных позже.

### Парный прогноз (полный, в MVP H2H)
- **Рейтинг пары для прогноза:** `(A+B)/2 + Bmax · (1 - 0.5^(S_partner / S0))`
- **Форма пары:** среднее индивидуальных парных форм партнёров.
- **pair-vs-pair H2H:** аналогично одиночному P3 (с учётом риска доступности данных).
- Далее — та же логистика и смешивание с P_h2h.

## 5. Сценарии бота (MVP — слой 1)

### Вход и навигация
- **Гибрид:** `/start` → главное меню (кнопки) + свободный ввод текста = поиск игрока.
- **Меню:** «Найти игрока», «Сравнить (H2H)», «Помощь».

### Поиск игрока
- Нечёткий поиск **от 3 символов** — по **нику или ФИО** (не обязательно оба поля).
- При нескольких совпадениях — список (до 5–10): ник, ФИО, город → пользователь выбирает.
- Не найден → подсказка (проверьте написание; игрок может отсутствовать в базе региона).
- **Индекс поиска:** упор на **неполную фамилию** — токены ФИО + триграммы (PostgreSQL `pg_trgm`); ник — дополнительное поле.

### Карточка игрока
- **Первый экран (компактно):** ник, ФИО, город, сводка рейтингов по разрядам, последний турнир.
- **Детали по кнопкам:** «Соперники», «H2H», «История рейтинга» (график PNG).
- **Дисциплина:** на карточке — сводка по всем разрядам; **выбор дисциплины только при H2H и прогнозе** (кнопки s/d/MD/WD/XD).
- **Актуальность:** внизу экрана — «Данные на DD.MM.YYYY».

### Соперники
- Топ по числу встреч + кнопка «Все соперники» с пагинацией.

### H2H (игрок A vs B)
- **Два входа:** из карточки («Сравнить с соперником») **и** команда `/h2h`.
- **`/h2h`:** поиск игрока A → поиск игрока B → выбор дисциплины → результат.
- **Экран результата (текст):** баланс W-L, последние матчи, прогноз (%), индекс S, форма обоих.

### График рейтинга
- Кнопка «История рейтинга» → **PNG на сервере** (JFreeChart или аналог) по данным профиля.

## 6. Подбор партнёров (слой 2, после MVP H2H)

### Сценарий
- **Цель:** найти партнёра **на конкретный будущий турнир**.
- **Вход:** кнопка «Найти партнёра» на экране турнира + пункт в меню (выбор турнира из ближайших).
- Пользователь указывает себя через поиск профиля (привязка TG↔игрок — позже).

### Пул кандидатов
- Игроки **нужного уровня** для данного турнира.
- **Ещё не заявлены** на этот турнир с другим партнёром (нет пары в регистрации).

### Фильтры (обязательные)
1. **Лимит рейтинга пары:** `(R_A + R_B) / 2 ≤ лимит турнира`. Лимит — из `<var>` названия/страницы (380, 500; «отк» = без лимита).
2. **Гео:** Москва и МО — **единый пул** (регион r77).
3. **Положительная совместная история** (для блока «Уже играли успешно»):
   - за последние **T месяцев** (конфиг): **суммарная дельта рейтинга обоих** в совместных турнирах **> 0**.

### Выдача
- **Два блока:**
  - «Уже играли успешно» — прошли фильтр совместной истории.
  - «Новые кандидаты» — подходят по уровню и гео, но без совместных турниров / без положительной истории.
- В каждой строке: ник, рейтинг, город, `(A+B)/2`, **score совместимости** (формула score — уточнить при реализации; база: близость к лимиту, дельта совместных турниров, S_partner).

## 7. Модель данных

### Сущности
- **Player** — ID (badminton4u), ник, ФИО, город, дата рождения, рука, рейтинги по дисциплинам, история рейтинга (таймлайн).
- **Tournament** — ID, дата, сырое название, **категория** (SE, DF, XDC… — парсим из названия/страницы), **лимит рейтинга** (`<var>`), коэффициент, клуб, город, статус (будущий/прошедший).
- **Participation** — игрок↔турнир: место, рейтинг до/дelta/после, баланс матчей и сетов; для пар — ссылка на Pair.
- **Pair** — player1, player2, discipline; совместные турниры (для S_partner и подбора партнёров).
- **Match** — до **4 игроков** (2+2 для пар), discipline, турнир, стадия, дата, счёт по сетам, дельты, длительность.
- **RivalSummary** — player↔opponent: W, L, delta (из `rivals`, храним при слепке).
- **SnapshotMeta** — дата последнего слепка региона.

### Хранение vs ленивая загрузка
| Данные | При слепке | По запросу |
|---|---|---|
| Турниры, участники, профили | ✓ | |
| Rivals (сводка) | ✓ | |
| Games (детальные матчи) | | ✓ |
| График рейтинга | данные из профиля | генерация PNG |

## 9. MVP (слой 1)

Поиск игрока по нику/ФИО → карточка игрока → **анализ H2H** (одиночка и пары):
расширенный текстовый вывод + прогноз P3 + индекс S + форма.
Привязки Telegram↔игрок **нет**.

**Следующий этап (слой 2):** подбор партнёра на турнир (раздел 6).
