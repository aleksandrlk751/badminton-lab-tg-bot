# Бриф проекта: Telegram-бот для любителей бадминтона (v2)

> Рабочий документ для продолжения обсуждения. Фиксирует принятые решения и открытые вопросы,
> чтобы можно было возобновить работу в новом чате.
> Статус: **реализация**, этапы **0–6 ✓** (подбор партнёра); **7** — H2H DoD. Источник/БД — [`data-model.md`](data-model.md).

## 1. Видение продукта

Бот-помощник для любительского бадминтона (комьюнити ЛАБ). Три слоя развития:

1. **Справочно-аналитический** (пассивный): игроки, турниры, история встреч, рейтинги, «сыгранность».
2. **Рекомендательный**: подбор партнёров, граф связанности игроков/турниров.
3. **Лайв-помощник** (активный): просчёт сетки на турнире, ведение счёта по партиям.

Развитие от слоя 1 к слою 3. Слои 1–2 питаются от внешних источников (парсинг), слой 3 генерирует
собственные данные во время игры.

**Roadmap (продукт):** v1 = слой 1 (H2H) → слой 2 (партнёры) → слой 3 (лайв) → VPS.  
**Порядок в коде** — см. [`PLAN.md`](PLAN.md) (партнёры **6** реализованы раньше DoD H2H **7**).  
**v2:** график рейтинга, граф связанности, `badminton77.ru`, расширение лайва.

## 3. Официальные формулы ЛАБ (зеркалим as-is, не выдумываем)

> Полный справочник формул и всех констант с дефолтами и локализацией в коде — [`FORMULAR.md`](FORMULAR.md).

- **Дельта рейтинга одиночки за матч:** `(100 - (R_поб - R_проигр)) / 10`.
  - Если рейтинг победителя ≥ рейтинга проигравшего на 100+ → дельта = 0 (победа «ожидаема»).
  - Симметрично: проигравший теряет столько же. Пол рейтинга = 1.
  - **Следствие:** разница ~100 пунктов ≈ решающее преимущество. Удобная шкала для прогноза.
- **Рейтинг пары:** `(A + B) / 2`, где A — рейтинг сильнейшего, B — слабейшего партнёра (источник: `content/10`).

## 4. Наши метрики (надстройки над данными)

### Индекс сыгранности S
```
S = Σ_i w(Δt_i)
```
`w(Δt)` — вес свежести (§2 `FORMULAR.md`: первый период `[0, H]` от `W_max` до `0.5` с показателем `α`,
далее `0.5^(Δt/H)`). Применяется к соперникам (H2H) и партнёрам (совместные парные турниры).
`H`, `W_max`, `α` — **параметры конфигурации**.

### Форма игрока
```
Form = Σ_i δ_i · w(Δt_i)
```
**Сумма** взвешенных свежестью дельт ЛАБ `δ_i` (знаковых). Дельта кодирует ожидаемость результата.

### Игровой акцент (карточка)

Две оси по парным MD/WD/XD (`PairCompositionService` + пол партнёра): **предпочтение** (доминирующий
тип + число игр за полгода) и **рекомендуемая категория** (сильная сторона — только тип пары;
δ и винрейт — в аналитике). Отдельные константы затухания. Формула — §2.7 `FORMULAR.md`.

### Стабильность игрока (0–100)

Показывает, насколько **предсказуемы** результаты относительно силы соперников — не путать с
**Формой** (направление: «в форме / не в форме»). Стабильно плохие и стабильно хорошие турниры
дают высокий балл; нестабильность — смешение знаков сюрпризов **внутри** турнира или **между**
соседними турнирами.

**Сюрприз матча:** `|δ| > ε` (порог `ε`, дефолт **11.5**). Парные — один `δ` на матч (как Form).

**Внутри турнира** `t` — считаем `n⁺`, `n⁻` (число матчей с `δ > ε` и `δ < −ε`):

```
S^within_t = 1                              если n⁺ = 0 или n⁻ = 0
S^within_t = 1 − min(n⁺, n⁻) / max(n⁺, n⁻)  иначе
```

**Тон турнира** (для межтурнирной проверки):

```
D_t = Σ δ_i  по матчам с |δ_i| > ε
```

`D_t > 0` — турнир сюрпризно хороший; `D_t < 0` — сюрпризно плохой; `D_t = 0` — нейтральный.

**Между соседними турнирами** (хронология по `starts_at`), переход `k`: `(t_{k−1} → t_k)`:

```
S^between_k = 0     если D_{t_{k−1}} · D_{t_k} < 0   (плохой → хороший или хороший → плохой)
S^between_k = 0.8    если D_{t_{k−1}} = 0  или  D_{t_k} = 0   (нейтральный турнир в паре)
S^between_k = 1     иначе
```

**Итог** — взвешенное среднее по турнирам и переходам (`w(Δt)` — та же кривая, что у S/Form;
`Δt` — от даты турнира):

```
Stability = 100 · ( Σ_t S^within_t · w(Δt_t) + Σ_k S^between_k · w(Δt_{t_k}) )
            / ( Σ_t w(Δt_t) + Σ_k w(Δt_{t_k}) )
```

Один турнир в истории → только `S^within`. Мало данных (нет матчей с `|δ| > ε`) — показатель не
выводим. Полный справочник — [`FORMULAR.md`](FORMULAR.md) §2.8.

**На карточке:** подпись «Стабильность», затем emoji зоны (без числа), строка после «Форма 👊»:

| Stability | Emoji |
|---|---|
| < 70 | 🔴 |
| 70 – 80 | 🟡 |
| 80 – 86 | ⚪ |
| 86 – 92 | 🟢 |
| ≥ 92 | 🔥 |

Пороги — `badminton-lab.metrics.stability-zones.*` (калибровка на слепке r77, ε=11.5, S=0.8).

### Прогноз P3 — явная формула

**Шаг 1. Эффективный рейтинг:**
```
R_eff_A = R_A + k · Form_A
R_eff_B = R_B + k · Form_B
```

**Шаг 2. Модель по рейтингу (логистика, масштаб 100):**
```
P_model = 1 / (1 + 10^(-(R_eff_A - R_eff_B) / 100))
```

**Шаг 3. Вероятность из личной истории (сглаживание Лапласа):**
```
P_h2h = (W_A + 1) / (W_A + W_B + 2)
```

**Шаг 4. Смешивание с весом индекса сыгранности:**
```
w = S / (S + S_ref)
P = (1 - w) · P_model + w · P_h2h
```

**Отображение:** «Фаворит: A (≈62%)» + краткое обоснование (рейтинг, форма, H2H, S).

Параметры `k`, `S_ref`, `H` — **в конфиге**, калибровка на реальных данных позже.

### Парный прогноз (полный, в MVP H2H)
- **Рейтинг пары для прогноза:** `(A+B)/2 + Bmax · (1 - 0.5^(S_partner / S0))`
- **Форма пары:** среднее индивидуальных парных форм партнёров.
- **pair-vs-pair H2H:** аналогично одиночному P3 (с учётом риска доступности данных).
- Далее — та же логистика и смешивание с P_h2h.

## 5. Сценарии бота (MVP — слой 1)

### Вход и навигация
- **Гибрид:** `/start` → главное меню (кнопки) + свободный ввод текста = поиск игрока.
- **Меню:** «Найти игрока», «Сравнить (H2H)», «🤝 Партнёр на турнир», «Помощь».

### Поиск игрока
- Нечёткий поиск **от 3 символов** — по **нику или ФИО** (не обязательно оба поля).
- При нескольких совпадениях — список (до 5–10): ник, ФИО, город → пользователь выбирает.
- Не найден → подсказка (проверьте написание; игрок может отсутствовать в базе региона).
- **Индекс поиска:** упор на **неполную фамилию** — токены ФИО + триграммы (PostgreSQL `pg_trgm`); ник — дополнительное поле.

### Карточка игрока
- **Первый экран (компактно):** ник, ФИО, город, сводка рейтингов по разрядам, **форма**, **игровой акцент**, **рекомендуемая категория**, последний турнир.
- **Игровой акцент:** две оси по парным матчам MD/WD/XD (одиночки вне акцента):
  - **Предпочтение** — тип пары с максимальной взвешенной долей игр (`w(Δt)` без жёсткого отсечения по времени);
    на экране — категория с заглавной буквы и на следующей строке число игр за последние 6 месяцев
    по доминирующему типу (заголовок с 📊).
  - **Рекомендуемая категория** (ось 2) — тип с максимальной взвешенной средней δ; на экране — категория
    с заглавной буквы (заголовок с 🧮); типичная δ и винрейт — в последующей аналитике (другие меню).
  Тип пары — `PairCompositionService` по полу игрока и партнёра. Формула — §2.7 `FORMULAR.md`.
- **Детали по кнопкам:** «Соперники», «H2H», «История рейтинга» (график PNG).
- **Дисциплина:** на карточке — сводка по всем разрядам; **выбор дисциплины только при H2H и прогнозе** (кнопки s/d/MD/WD/XD).
- **Актуальность:** внизу экрана — «Данные на DD.MM.YYYY».

### Соперники
- Топ по числу встреч + кнопка «Все соперники» с пагинацией.

### H2H (игрок A vs B)
- **Два входа:** из карточки («Сравнить с соперником») **и** команда `/h2h`.
- **`/h2h`:** поиск игрока A → поиск игрока B → выбор дисциплины → результат.
- **Экран результата (текст):** баланс W-L, последние матчи, прогноз (%), индекс S, форма обоих.
- **Тип пары (парный H2H):** по `player.sex` двух игроков на каждой стороне — однополая мужская (MD),
  однополая женская (WD), микст (XD) или «пол неизвестен»; метка на экране + согласование с выбранным разрядом
  (предупреждение при несоответствии). Generic `D` — подсказка фактического типа, если пол известен.
  Логика — `PairCompositionService` (этап 5), потребление — **этап 7** (DoD H2H).

### График рейтинга
- Кнопка «История рейтинга» → **PNG на сервере** (JFreeChart или аналог) по данным профиля.
- **v2** — в v1 кнопка остаётся заглушкой; данные `player_rating_history` уже в слепке.

## 6. Подбор партнёров (слой 2, этап 6 ✓)

### Сценарий
- **Цель:** найти партнёра **на конкретный будущий турнир**.
- **Вход:** кнопка «Найти партнёра» на экране турнира + пункт в меню (выбор турнира из ближайших).
- Пользователь: **привязанный профиль** («Мой профиль», этап 8) или поиск «я» вручную (сейчас — этап 6).

### Пул кандидатов
- Игроки **нужного уровня** для данного турнира.
- **Ещё не заявлены** на этот турнир с другим партнёром (нет пары в регистрации).

### Фильтры (обязательные)
1. **Лимит рейтинга пары:** `(R_A + R_B) / 2 ≤ лимит турнира`. Лимит — из `<var>` названия/страницы (380, 500; «отк» = без лимита).
2. **Гео:** Москва и МО — **единый пул** (регион r77).
3. **Пол / тип пары:** по разряду турнира и `player.sex` — MD → партнёр `M`, WD → `F`, XD → противоположный пол пользователю;
   в выдаче — метка типа будущей пары («мужская пара», «микст» и т.д., `PairCompositionService`).
4. **Успешная совместная история** (для приоритета в блоке «Уже играли» и ⭐):
   - за последние **T месяцев** (конфиг): **суммарная дельта рейтинга обоих** в совместных турнирах **> 0**.

### Выдача
- **Два блока:**
  - «Уже играли» — есть хотя бы один совместный турнир в паре; выше в списке — с положительной историей за `T` мес.
  - «Новые кандидаты» — подходят по уровню и гео, **без** совместных турниров в истории.
- В каждой строке: ник, рейтинг, город, `(A+B)/2`, **score совместимости** (0–100, см. ниже).
- **Сортировка:** внутри блока — сначала кандидаты с **совпадением рекомендуемой категории** (игровой акцент,
  `GameAccentService.preferenceType`) и разряда турнира (MD/WD/XD), затем по score убыванию; в строке 🎯 / ⭐.
- **Идеальный партнёр (⭐):** подходит по среднему рейтингу и лимиту, **не** заявлен на турнир с другим,
  **положительная форма**, **играли вместе** с положительной историей за `T` мес., **категория акцента = турнир**.

### Score совместимости (0–100)

Компоненты (0–1, кроме `C_form` ∈ [−1, 1]):

```
C_limit = min(C_pair, C_player)                   # см. FORMULAR §2.5
C_delta = sigmoid(Δ_joint_w / D_scale)            # Δ_joint_w ≤ 0 → 0
C_S     = S_partner / (S_partner + S_ref_partner)
C_form  = Form кандидата (§2.2), нормировка §2.5 FORMULAR; при Form &gt; 0 умножение на k_stab по зоне §2.8
C_accent = sigmoid(δ_cat / D_scale)               # δ_cat ≤ 0 или нет матчей типа турнира → 0
```

`δ_cat` — взвешенная средняя δ по матчам **разряда турнира** (MD/WD/XD), §2.7; не привязана к «рекомендуемой» категории на карточке.

```
score = clamp(100 · (w1·C_limit + w2·C_delta + w3·C_S + w4·C_form + w5·C_accent), 0, 100)
```

`w1`…`w5`, `D_scale`, `S_ref_partner`, `F_form`, `T` — **в конфиге** (`T` — сортировка/⭐ в bot).
Для «Новых кандидатов» без совместной истории: `C_delta = 0`, `C_S = 0`.

## 6.1 Привязка Telegram ↔ игрок (v1, этап 8)

- **1 TG-аккаунт ↔ 1 профиль** badminton4u (player id).
- Вход: «Мой профиль» / `/me` → поиск + подтверждение или карточка при уже привязанном игроке.
- Используется: подбор партнёра (подстановка «я»), лайв-помощник (кто ведёт матч).
- Отвязка явная; чужую привязку не перезаписываем без подтверждения.

## 6.2 Лайв-помощник (v1 MVP, этап 9)

- **Слой 3:** собственные данные бота во время игры (не парсинг).
- MVP: выбор/создание матча, **счёт по сетам** (2:0, 2:1); счёт по очкам в партии — недоступен на источнике.
- Полная сетка турнира, `badminton77.ru`, офлайн-sync — **v2** (расширение после деплоя).
- Требует привязки TG↔игрок (§6.1).
