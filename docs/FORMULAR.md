# Формулы и константы (FORMULAR)

> Единый справочник формул проекта и их дефолтных констант: **где применяется** (локализация в
> коде/конфиге) и **что означает**.
>
> Источник **смысла** формул — [`BRIEF.md`](BRIEF.md) (§3 официальные, §4 метрики, §6 подбор).
> Этот документ фиксирует **локализацию и дефолты**. При изменении формул/констант —
> синхронизируй `BRIEF.md` и этот файл.

## Принципы

1. **Официальные формулы ЛАБ зеркалим as-is** — не заменяем своими (правило проекта, `AGENTS.md`).
2. **Собственные метрики** (S, форма, прогноз, скор) — надстройка сверху; их константы **вынесены в
   конфиг** и подлежат **калибровке позже** (открытый вопрос, `BRIEF.md` §11).
3. Все числовые дефолты метрик живут в одном месте: `core/src/main/resources/application-core.yml`
   (`badminton-lab.metrics.*`) и связаны с `MetricsProperties`.

## 1. Официальные формулы ЛАБ (as-is)

### 1.1 Дельта рейтинга одиночки за матч

```
δ = (100 - (R_поб - R_проигр)) / 10
```

- Если рейтинг победителя ≥ рейтинга проигравшего на **100+** → `δ = 0` (победа ожидаема).
- Проигравший теряет столько же (симметрия). **Пол** рейтинга = 1.

**Локализация:** не пересчитываем. Значение дельты берём из источника при парсинге и сохраняем:
- `TournamentResultsParser` → `ResultUpsertService` → `participation.rating_delta`;
- `TournamentGamesParser` → `MatchUpsertService` → `match_player.rating_delta`
  (для пары дельта **одинакова** для обоих игроков стороны).

**Описание:** знаковая дельта кодирует ожидаемость результата — база для метрики «Форма» (§2.2).

### 1.2 Рейтинг пары

```
R_пара = (A + B) / 2
```

где `A` — сильнейший, `B` — слабейший партнёр. Источник: `content/10`.

**Локализация:** `PairRatingService#pairRating` (`core.metrics`, **реализовано, этап 3**); в подборе
партнёров (§2.5) — `PartnerScoreService` (**планируется, этап 7**).

## 2. Наши метрики (надстройка)

Обозначения: `Δt_i` — время с i-й встречи; `H` — период полураспада; `δ_i` — знаковая дельта ЛАБ;
`W_max` — потолок веса в начале первого периода; `α` — крутизна спада в `[0, H]`.

**Вес свежести** (общий для §2.1 и §2.2, `MetricMath#decayWeight`):

```
         ⎧ W_max + (0.5 − W_max) · 2 · (1 − 0.5^((Δt/H)^α))     если Δt ≤ H
w(Δt) =  ⎨
         ⎩ 0.5^(Δt/H)                                           если Δt > H
```

При `W_max = 1` и `α = 1` — прежняя кривая `0.5^(Δt/H)` на всём диапазоне.
Якоря `w(H) = 0.5`, `w(2H) = 0.25`, … — без изменений.

### 2.1 Индекс сыгранности S

```
S = Σ_i w(Δt_i)
```

**Константы:** `H`, `W_max`, `α`. **Применяется к** соперникам (H2H) и партнёрам (совместные турниры).
**Локализация:** `PlayabilityIndexService` (`core.metrics`, **реализовано, этап 3**). **Описание:**
«вес» накопленного опыта встреч с затуханием по каждой встрече.

### 2.2 Форма игрока

```
Form = Σ_i δ_i · w(Δt_i)
```

**Константы:** `H`, `W_max`, `α`. **Локализация:** `FormService` (`core.metrics`, **реализовано, этап 3**), вход —
`RatingDeltaEvent` из `match_player.rating_delta`. **Описание:** сумма взвешенных свежестью знаковых
дельт — «в форме / не в форме».

### 2.3 Прогноз P3 (одиночный)

```
R_eff_A = R_A + k · Form_A
R_eff_B = R_B + k · Form_B
P_model = 1 / (1 + 10^(-(R_eff_A - R_eff_B) / 100))     # логистика, масштаб 100
P_h2h   = (W_A + 1) / (W_A + W_B + 2)                    # сглаживание Лапласа
w       = S / (S + S_ref)
P       = (1 - w) · P_model + w · P_h2h
```

**Константы:** `k`, `S_ref` (и `H` внутри `Form`/`S`). Масштаб `100` — из официальной шкалы (§1.1),
**не параметр**. **Локализация:** `ForecastService` (`core.metrics`, **реализовано, этап 3**),
результат — `ForecastResult`. **Описание:** смесь модельной вероятности (по эффективному рейтингу) и
личной истории; вес смеси растёт со сыгранностью `S`.

### 2.4 Парный прогноз

```
R_пара_прогноз = (A + B) / 2 + Bmax · (1 - 0.5^(S_partner / S0))
Form_пара      = среднее индивидуальных парных форм партнёров
```

Далее — та же логистика и смешивание с `P_h2h`, что в §2.3.

**Константы:** `Bmax`, `S0` (плюс `k`, `S_ref`, `H`). **Локализация:**
`PairRatingService#pairRatingForForecast` + `ForecastService` (`core.metrics`, **реализовано,
этап 3**). **Описание:** к среднему рейтингу пары добавляется бонус за сыгранность партнёров,
насыщающийся к `Bmax`.

### 2.5 Score совместимости партнёра (0–100)

```
C_limit    = min(1, ((R_A + R_B) / 2) / limit)     # ближе к лимиту (не выше) — лучше
C_delta    = sigmoid(Δ_joint / D_scale)            # Δ_joint — суммарная дельта обоих за T мес.
C_S        = S_partner / (S_partner + S_ref_partner)
score_base = 100 · (w1·C_limit + w2·C_delta + w3·C_S)
score      = score_base · 1.2                      # boost только в блоке «Уже играли успешно»
```

Для «Новых кандидатов» без совместной истории: `C_delta = 0`, `C_S = 0` (score ≈ от `C_limit`).

**Константы:** `w1`, `w2`, `w3`, `D_scale`, `S_ref_partner`, `T`, boost `1.2`.
**Локализация:** `PartnerScoreService` (**планируется, этап 7**). **Описание:** нормированная оценка
кандидата в партнёры на конкретный турнир (лимит рейтинга + совместная история + сыгранность).

## 3. Константы метрик — дефолты и локализация

Все ключи — под `badminton-lab.metrics.*` в
[`core/src/main/resources/application-core.yml`](../core/src/main/resources/application-core.yml),
связаны с полями
[`MetricsProperties`](../core/src/main/java/ru/badmintonlab/core/config/MetricsProperties.java).

| Символ | Дефолт | Ключ конфига | Поле `MetricsProperties` | Применяется (§) | Описание |
|---|---|---|---|---|---|
| `H` | `180` | `half-life-days` | `halfLifeDays` | 2.1, 2.2, 2.3, 2.4 | Период полураспада (дни) для `S` и `Form` |
| `W_max` | `0.8` | `early-decay-max` | `earlyDecayMax` | 2.1, 2.2 | Потолок веса свежести в начале первого периода `(0.5, 1]` |
| `α` | `0.5` | `early-decay-power` | `earlyDecayPower` | 2.1, 2.2 | Крутизна спада веса в первом периоде (`> 0`; `1` = прежняя форма кривой) |
| `k` | `0.5` | `form-k` | `formK` | 2.3, 2.4 | Вклад формы в эффективный рейтинг |
| `S_ref` | `1.0` | `s-ref` | `sRef` | 2.3, 2.4 | Опорная сыгранность для веса смешивания `w` |
| `Bmax` | `20.0` | `b-max` | `bMax` | 2.4 | Максимальный бонус парного рейтинга за сыгранность |
| `S0` | `1.0` | `s0` | `s0` | 2.4 | Масштаб насыщения бонуса пары |
| `w1` | `0.4` | `w1` | `w1` | 2.5 | Вес компонента `C_limit` в score |
| `w2` | `0.3` | `w2` | `w2` | 2.5 | Вес компонента `C_delta` в score |
| `w3` | `0.3` | `w3` | `w3` | 2.5 | Вес компонента `C_S` в score |
| `D_scale` | `10.0` | `d-scale` | `dScale` | 2.5 | Масштаб сигмоиды для суммарной дельты `Δ_joint` |
| `T` | `12` | `partner-history-months` | `partnerHistoryMonths` | 2.5 | Окно совместной истории (месяцы) |

> Инварианты (`MetricsProperties`): `halfLifeDays > 0`, `earlyDecayMax ∈ (0.5, 1]`, `earlyDecayPower > 0`,
> `partnerHistoryMonths > 0`.
> Ожидается `w1 + w2 + w3 = 1` (сейчас `0.4 + 0.3 + 0.3`), но это **не** валидируется в коде.

## 5. Калибровка

Дефолты метрик (`H`, `k`, `S_ref`, `Bmax`, `S0`, `w1..w3`, `D_scale`, `T`) — **стартовые
плейсхолдеры**. Калибровка на реальных данных и A/B на эталонах — отдельная задача roadmap
(`PLAN.md` v2+, `BRIEF.md` §11). До калибровки менять значения только через конфиг, без правок
формул.
